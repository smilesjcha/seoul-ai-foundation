# 강의 운영 런북

## 전체 운영 원칙

- 모든 세션은 "문서 → 구현 → 검토" 순서를 유지합니다.
- 수강생이 막히는 지점은 대부분 "무엇을 요청해야 할지 모름"이므로, 요청 예시를 계속 보여줍니다.
- Claude Desktop은 정답 생성기가 아니라 작업 파트너라는 점을 반복 강조합니다.

## 세션별 운영 포인트

### 13:00-13:30 환경 세팅 & OT

- 저장소 구조를 먼저 보여줍니다.
- 오늘의 최종 데모 화면을 먼저 보여주면 집중도가 올라갑니다.
- Claude에게 프로젝트 설명을 시켜보며 "문맥 읽히기"를 시연합니다.

### 13:30-14:00 문제 정의

- "자동화할 수 있는 일"보다 "반복되고 기준이 있는 일"을 먼저 찾는 법을 설명합니다.
- 민원/FAQ 자동화가 왜 좋은 출발점인지 사례와 함께 안내합니다.

### 14:00-14:30 PRD

- PRD는 거창한 문서가 아니라 AI와 팀이 공유할 기준 문서라는 점을 강조합니다.
- 범위 제한이 없는 PRD는 AI 협업 품질도 떨어진다는 점을 설명합니다.

### 14:40-15:10 설계 문서

- UI보다 데이터 흐름을 먼저 그립니다.
- API, 상태, 예외 케이스를 간단히 정리하게 합니다.

### 15:10-16:40 프로토타입 구현

- 백엔드와 프론트엔드를 동시에 만들기보다 API부터 완성합니다.
- 수강생에게 Claude 요청을 작은 단위로 나누게 합니다.
- 예시:
  - "문의 목록 API 먼저 만들어줘"
  - "상세 조회에 추천 결과 포함해줘"
  - "UI는 목록/상세 2단 레이아웃으로 구현해줘"

### 16:40-17:10 품질 개선

- 잘못된 분류 예시를 일부러 만들어 보여줍니다.
- 톤, 누락, 과잉 응답을 어떻게 점검할지 다룹니다.

### 17:10-17:30 리뷰 & 테스트

- Claude/Codex에게 리뷰를 요청할 때는 "버그, 회귀, 리스크 중심"으로 요청하게 합니다.
- 테스트는 모든 걸 자동화하기보다 핵심 시나리오부터 잡습니다.

## 라이브 데모 팁

- 인터넷이나 설치 이슈에 대비해 Mock 데이터 기반으로도 전 과정이 보이게 준비합니다.
- 화면이 깨져도 문서와 API만 살아 있으면 수업은 계속 진행할 수 있습니다.
- 어려운 질문이 나오면 "오늘의 범위"와 "실무 확장 포인트"를 분리해 답하면 좋습니다.
