# PRD Review Perspectives

## 목적

이 문서는 AI 개발자 인턴들이 PRD를 단순 요구사항 문서가 아니라, 회사 관점, 사용자 경험 관점, 엔지니어링 관점에서 함께 검토하는 기준으로 사용할 수 있도록 만든 리뷰 가이드입니다.

핵심 메시지:

> 좋은 PRD는 기능을 많이 적은 문서가 아니라, 서로 다른 역할의 사람이 같은 방향으로 판단할 수 있게 만드는 문서입니다.

## 1. Leadership / 임원 관점

### 관점의 핵심

회사를 위해 이 기능을 왜 해야 하는지, 투자할 가치가 있는지, 리스크를 감당할 수 있는지 판단합니다.

### 리뷰 질문

| 질문 | 확인 기준 |
| --- | --- |
| 왜 지금 이 문제를 풀어야 하는가 | 반복 업무, 비용, 품질 편차, 고객 경험 문제가 명확한가 |
| 회사에 어떤 가치가 있는가 | 시간 절감, 처리량 증가, 품질 표준화, 운영 리스크 감소와 연결되는가 |
| MVP 범위가 적절한가 | 빠르게 검증 가능한 크기로 줄어 있는가 |
| 리스크가 보이는가 | 개인정보, 정책 오류, 자동 발송 위험이 명시되어 있는가 |
| 성공 기준이 의사결정에 쓸 수 있는가 | 정성적 만족이 아니라 운영 지표와 연결되는가 |

### 평가 예시

- 좋은 PRD: "운영자 초안 작성 시간을 줄이고, FAQ 기반 응답 품질 편차를 줄이는 것을 목표로 한다."
- 아쉬운 PRD: "AI로 민원을 자동화한다."

## 2. UI/UX Designer 관점

### 관점의 핵심

PRD가 실제 화면으로 옮겨졌을 때 사용자가 헷갈리지 않고, 필요한 판단을 빠르게 할 수 있는지 봅니다.

### 리뷰 질문

| 질문 | 확인 기준 |
| --- | --- |
| 주요 사용자는 누구인가 | 운영자, 시민, 관리자 중 이번 화면의 1차 사용자가 분명한가 |
| 사용자가 다음 행동을 알 수 있는가 | 목록, 상세, 응답 초안, 검토 포인트의 흐름이 자연스러운가 |
| 중요한 정보가 먼저 보이는가 | 문의 원문, 카테고리, FAQ 매칭, 검토 포인트 우선순위가 맞는가 |
| AI 결과를 맹신하지 않게 설계되어 있는가 | 초안과 검토 포인트가 분리되어 보이는가 |
| Edge case 화면이 고려되어 있는가 | FAQ 미매칭, 담당자 확인 필요, 오류 상태가 표현되는가 |

### 평가 예시

- 좋은 PRD: "운영자는 문의 상세에서 원문, 추천 FAQ, 응답 초안, 검토 포인트를 한 화면에서 확인한다."
- 아쉬운 PRD: "AI 답변을 화면에 보여준다."

## 3. Engineer 관점

### 관점의 핵심

PRD를 기준으로 백엔드, 프론트엔드, 데이터, AI 로직을 어떻게 나누어 구현할 수 있는지 봅니다.

### 리뷰 질문

| 질문 | 확인 기준 |
| --- | --- |
| 데이터 모델이 보이는가 | Inquiry, FAQ, Recommendation 같은 핵심 객체가 분명한가 |
| API 경계가 보이는가 | 목록, 상세, 추천 생성 API로 나눌 수 있는가 |
| 프론트엔드 상태가 예측 가능한가 | 선택된 문의, 로딩, 오류, 추천 결과 상태가 정의 가능한가 |
| AI 로직이 분리 가능한가 | 분류, FAQ 매칭, 응답 생성, 검토 포인트 생성을 나눌 수 있는가 |
| 테스트할 수 있는가 | 정상, 미매칭, 없는 id, 민감 문의 같은 케이스가 있는가 |

### 평가 예시

- 좋은 PRD: "문의 상세 API는 inquiry와 recommendation을 함께 반환한다."
- 아쉬운 PRD: "AI가 적절한 답변을 만든다."

## 4. 세 관점 통합 리뷰 매트릭스

| 평가 영역 | Leadership | UI/UX Designer | Engineer |
| --- | --- | --- | --- |
| 문제 정의 | 회사가 풀 가치가 있는가 | 사용자의 실제 불편인가 | 구현 가능한 문제인가 |
| MVP 범위 | 검증 가능한 크기인가 | 핵심 화면이 충분한가 | API와 데이터가 단순한가 |
| 성공 기준 | 의사결정에 쓸 수 있는가 | 사용자 행동으로 확인 가능한가 | 로그와 테스트로 확인 가능한가 |
| 리스크 | 조직 리스크가 보이는가 | 사용자 오해를 줄이는가 | 예외 처리가 가능한가 |
| 다음 액션 | 투자/확장 판단 가능 | 화면 설계 가능 | 구현 계획 가능 |

## 5. Claude 리뷰 요청 프롬프트

```md
docs/04-prd-faq-automation.md를 아래 세 관점으로 리뷰해줘.

1. Leadership / 임원 관점:
- 회사 가치
- 운영 효율
- 리스크
- 성공 기준

2. UI/UX Designer 관점:
- 화면 흐름
- 사용자 판단 가능성
- AI 결과를 검토 가능한 방식으로 보여주는지
- Edge case 화면

3. Engineer 관점:
- API 경계
- 데이터 모델
- 프론트엔드 상태
- AI 로직 분리
- 테스트 가능성

각 관점별로 좋은 점, 부족한 점, 바로 고칠 제안을 표로 정리해줘.
```

## 6. 수업 중 운영 방법

- 4명의 AI 개발자 인턴이 있다면 각자 하나의 관점을 맡깁니다.
- 1명은 Leadership 관점, 1명은 UI/UX Designer 관점, 1명은 Engineer 관점, 1명은 전체 정리자 역할을 맡습니다.
- 각자 5분 리뷰 후, 5분 공유합니다.
- 최종적으로 PRD v2에 반영할 항목 3개만 고릅니다.
